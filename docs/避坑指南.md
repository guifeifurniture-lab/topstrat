## ⚠️ 四大维度避坑

### 1. 任务复杂度

| ✅ 建议 | ❌ 避免 |
|--------|--------|
| 多步骤复杂任务 | 简单任务复杂化 |
| 需要多角色协作 | 单 LLM 可调用的任务 |
| 有明确输入输出 | 模糊开放的任务 |

#### 正确示例

```
✅ 竞品分析报告
   输入：竞品 URL
   流程：搜索 → 分析 → 撰写 → 审核
   输出：结构化报告

✅ 内容工厂
   输入：主题关键词
   流程：研究 → 撰写 → 编辑
   输出：博客文章

✅ 客户支持分诊
   输入：用户工单
   流程：分类 → 解决/升级
   输出：回复草稿/人工介入
```

#### 错误示例

```
❌ "写个笑话"
   → 直接调用单个 LLM 即可

❌ "今天天气如何"
   → 简单查询，无需多 Agent

❌ "帮我思考人生意义"
   → 模糊开放，无法结构化
```

---

### 2. 成本控制

| ✅ 建议 | ❌ 避免 |
|--------|--------|
| 开启缓存（不常变动数据） | 重复调用 LLM |
| 使用串行模式（新手） | 盲目使用分层模式 |
| 按需调用相关工程师 | 全员广播 |
| 传递文件路径 | 传递大量原始数据 |

#### 成本对比示例

**场景：** 竞品分析（5 个竞品页面）

| 方案 | Token 消耗 | 成本 | 推荐度 |
|------|-----------|------|--------|
| 串行 + 缓存 | ~50K | $0.01 | ✅ 推荐 |
| 串行无缓存 | ~100K | $0.02 | 🟡 可接受 |
| 分层模式 | ~300K | $0.06 | ❌ 避免（新手） |
| 全员广播 | ~500K | $0.10 | ❌ 严禁 |

#### 降本措施

```markdown
### 缓存策略
- 公司介绍 → 缓存 24 小时
- 竞品基础信息 → 缓存 7 天
- 历史分析报告 → 永久缓存（除非更新）

### 按需调用
- 网站开发任务 → 仅调用网站开发组
- AI 研究任务 → 仅调用 AI 研究组
- 避免全员广播

### 数据传递优化
- 大文件 → 传递路径，让下一个 Agent 读取
- 长文本 → 分段处理，避免上下文过长
```

---

### 3. 数据传递

| ✅ 建议 | ❌ 避免 |
|--------|--------|
| 小数据直接传递 | 大数据直接传递 |
| 大数据传递文件路径 | 在 Agent 间传递大量原始数据 |
| 使用 context 参数 | 上下文过长（>100K tokens） |

#### 正确示例

```markdown
### 小数据（直接传递）
任务 A 输出：{"status": "success", "count": 5}
任务 B 输入：直接使用

### 大数据（传递路径）
任务 A 输出：`reports/竞品数据 -2026-03-18.md`
任务 B 输入：读取文件路径
```

#### 错误示例

```markdown
### 错误：传递大量原始数据
任务 A 输出：{完整 HTML 内容 50KB...}
任务 B 输入：接收 50KB 数据

问题：
- Token 消耗翻倍
- 处理速度变慢
- 容易超出上下文限制
```

---

### 4. 实时性

| ✅ 建议 | ❌ 避免 |
|--------|--------|
| 异步处理（耗时任务） | 同步阻塞主线程 |
| 接受合理延迟 | 期待毫秒级响应 |
| 后台执行 + 通知 | 用户等待执行完成 |

#### 延迟预期

| 任务类型 | 预期延迟 | 说明 |
|---------|---------|------|
| 简单查询 | 1-3 秒 | 单次 LLM 调用 |
| 串行任务（3 步） | 10-30 秒 | A→B→C 顺序执行 |
| 复杂报告 | 1-5 分钟 | 多步骤 + 网络请求 |
| 批量处理 | 5-30 分钟 | 多个项目并行 |

#### 异步处理示例

```markdown
### 同步（避免）
用户发起任务 → 等待执行 → 显示结果
问题：用户需要等待

### 异步（推荐）
用户发起任务 → 立即确认 → 后台执行 → 完成后通知
优势：用户无需等待，可同时处理其他任务
```

## 🔧 错误处理与重试机制

### 常见错误类型

| 错误类型 | 原因 | 处理方式 |
|---------|------|---------|
| API 失败 | 网络问题/服务不可用 | 重试 3 次（指数退避） |
| 超时 | 任务执行时间过长 | 延长超时 + 通知 |
| 输出格式错误 | LLM 未遵循模板 | 重新生成（最多 3 次） |
| 上下文过长 | 输入数据太大 | 分段处理/摘要压缩 |
| Token 超限 | 超出模型限制 | 截断/分批处理 |

### 重试机制实现

```python
# 伪代码示例
def execute_with_retry(agent, task, max_retries=3):
    """
    带重试的执行函数
    """
    for attempt in range(max_retries):
        try:
            result = agent.execute(task)
            
            # 验证输出格式
            if not validate_output(result):
                raise ValueError("输出格式错误")
            
            return result
            
        except Exception as e:
            if attempt == max_retries - 1:
                # 最后一次重试失败
                log_error(e)
                notify_admin()
                raise
            
            # 指数退避等待
            wait_time = 2 ** attempt  # 1s, 2s, 4s
            sleep(wait_time)
```

### 谋士天团错误处理分工

| 错误类型 | 负责人 | 处理方式 |
|---------|--------|---------|
| API 失败 | 鲁班 | 重试 + 日志记录 |
| 超时 | 商鞅 | 延长超时 + 通知诸葛亮 |
| 格式错误 | 萧何 | 重新生成 + 模板优化 |
| 上下文过长 | 管仲 | 分段处理/摘要 |
| Token 超限 | 张良 | 截断/分批 |

## 🚨 常见陷阱与解决方案

### 陷阱 1：任务定义模糊

**问题：**
```
❌ "优化网站"
❌ "改进用户体验"
```

**解决：**
```
✅ "将首页加载时间从 3s 降低到 1.5s"
✅ "将 Lighthouse 性能分数从 70 提升到 90"
✅ "修复所有 P0 级别的 bug"
```

---

### 陷阱 2：上下文爆炸

**问题：**
- 每个任务都传递完整历史
- 多次迭代后上下文超过 100K tokens

**解决：**
```markdown
### 摘要策略
- 每 10 次交互生成一次摘要
- 只保留关键决策和结论
- 删除中间过程数据

### 文件引用
- 大文件存为独立文件
- 传递文件路径而非内容
- 需要时再读取
```

---

### 陷阱 3：过度工程化

**问题：**
```
简单任务：写一封邮件
❌ 配置 5 个 Agent + 分层管理 + 复杂工作流
✅ 直接调用孔子（内容工程师）即可
```

**解决：**
```markdown
### 任务复杂度评估
1. 是否需要多角色协作？否 → 单 Agent
2. 是否需要多步骤？否 → 单次调用
3. 是否需要持久化记忆？否 → 无状态

只有 3 个都是"是"，才使用完整工作流
```

---

### 陷阱 4：忽略 Token 成本

**问题：**
- 不监控 Token 消耗
- 月底发现成本远超预期

**解决：**
```markdown
### 成本监控
- 每日 Token 消耗报表（范蠡负责）
- 单次任务成本估算
- 设置预算告警（超过$10/天通知）

### 优化措施
- 优先使用便宜模型（qwen3.5-plus）
- 复杂任务才用昂贵模型
- 缓存命中不重复调用
```
